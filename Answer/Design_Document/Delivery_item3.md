1. 1:1 工单生成的“派单雪崩”陷阱（对应草稿第 4 节）

* **问题是什么**：
    草稿提出“所有告警 1:1 生成工单，立即进入派单队列”，完全没有任何降噪与合并机制。
* **带来严重后果**：
    违背了物联网运维的基本常识。物理传感器在临界点极易发生“脉冲震荡”，真实的灾害也会引发级联告警。如果 1:1 派单，系统会瞬间生成超多个独立工单，几十名工人的手机会被相同的派单指令疯狂轰炸，导致系统并发瘫痪与一线的“警报疲劳”。

* **怎么改**：
    必须在事件流接入后、派发前，强行插入“多级确定性降噪漏斗”。利用 Redis 锁进行单设备的 TTL 防抖（拦截震荡），并利用空间哈希算法将同一时间窗口内、同一楼栋/楼层的同源告警，强制折叠成一个唯一的“复合灾害包”后再向下游派发。

2. 大模型强行卡入核心调度主链路（对应草稿第 3、8 节）
* **问题是什么**：
    将日均 5,000+ 条的多源事件全部送入 LLM，依靠大模型来判定优先级并作为调度的唯一大脑。

* **带来严重后果**：
    - LLM 接口的秒级延迟绝对无法满足电梯困人等P0事件“毫秒级派发”的底线。让具有概率性“幻觉”特征的生成式AI来判断灾情，极有可能将罕见的火灾误判为 P3，引发不可挽回的生命财产损失。日均上万次的 API 强依赖，一旦阿里云网络抖动，整个园区的调度将彻底瘫痪。

* **怎么改**：
    - 实施“快慢双轨”物理隔离。IoT与视觉等结构化数据，直接走规则引擎快轨。LLM 被严格降级为“文本翻译器”，仅用于慢轨中处理非结构化的 App/微信投诉，输出标准 JSON 后立即交出控制权。

3. 异构运力物理死锁与“优先级灾难倒挂”（对应草稿第 6、7 节）
* **问题是什么**：
- 草稿对工人和机器人采用完全相同的贪心就近逻辑，并且在其自创的优先级公式中，将“投诉次数（complaint_count）”作为乘数因子进行绝对加权。
* **带来严重后果**：
    - 如果只看直线距离，系统极易将 B1 栋 3 楼的漏水任务，派给直线距离最近、但身处 B2 栋 3 楼的轮式机器人。机器人若无法跨域处理问题，将陷入原地打转的物理死锁。按照其公式 complaint_count * 10 + severity，一个有 50 个业主在微信群吐槽的“园区草坪灯不亮（P3，severity=1）”得分高达 501 分；而一个无人围观但极其致命的“地下室机房烟感报警（P0，severity=10）”得分仅为10分。维修人员将被迫先去修草坪灯，任由机房烧毁。
* **怎么改**：
    算法模型必须切割。对于优先级，摒弃自创的线性公式，实行“P0/P1 绝对抢占中断 + 剩余工单按 SLA 倒计时（ETA）排序”。对于异构运力，在调度数学模型中强制加入“空间防区隔离”硬约束；机器人脱离即时响应流，转入按区域锚定的L3定时批处理队列。
