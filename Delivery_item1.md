1. 异构运力的空间约束与调度边界
- 澄清点：园区共20栋楼但仅有3台轮式机器人，是否有跨楼层/跨栋物理机动能力？需求中要求“工人和机器人使用同一套分派逻辑”，是否意味着系统需假设机器人具备与人员同等的可达性？
- 架构考量：若强行纳入全局统筹，必然导致无干预能力的运力被就近误派，引发物理死锁。系统底层需实施运力硬隔离，将机器人剥离出响应式派单池，限定在其常驻空间内执行计划性巡检任务。

2. P95 延迟底线与 LLM 算力旁路
- 澄清点：日均 5,000 条告警多为结构化传感数据，SLA 要求苛刻。需求中要求全量过 LLM。是否允许系统引入分层路由，使结构化传感器数据直接走底层规则引擎，仅让非结构化文本（约 50 条/天）过大模型？
- 架构考量：大模型 API 推理存在秒级延迟与单点故障风险。采用快慢双轨制是保障P0事件5分钟内下发的核心。全量LLM处理不仅造成Token浪费，更会导致紧急安全事件出现严重的排队拥塞。

3. 频次权重与安全 SLA 矩阵的优先级冲突
- 澄清点：需求 3 提出“反映次数越多优先级越高”。当高频客诉（如空调损坏）与低频潜在高危事件（如配电房异味）同时并发时，“频次优先”是否绝对服从于既定的 SLA 矩阵？
- 架构考量：单纯的频次累加会导致调度系统的“饥饿现象”。派单打分函数需调整为“基准严重度为主、频次增益为辅”的复合算法，并赋予P0级灾害绝对抢占权。

4. 稀缺运力的 VRP 折损与调度中断
- 澄清点：需求中要求“识别到污渍立即派机器人清扫”。鉴于机器人已被限定在特定物理空间，若其正在执行该区域内的例行清扫路线，此时区域内新增污渍告警，是立即中断当前任务，还是允许将其降级为顺路航点？
- 架构考量：即便在封闭的单层空间内，频繁的即时中断也会导致底盘路径规划产生严重的“往复震荡”，极大折损有效工作时长与电池续航。调度核心应引入“计划性蓄水池”，将P3级保洁任务设为非抢占事件，通过算法重算局部最优路径，实施顺路核销。

5. 运力极值下的系统熔断与人工兜底
- 澄清点：需求中要求“尽量全自动”。当极端场景下在岗运力负荷打满，且仍有紧急告警涌入时，系统是继续排队分派，还是触发熔断机制由调度人员介入？
- 架构考量：涉及高并发下的服务降级。必须在系统状态机中引入全局运力水位监控，一旦超过安全阈值则降级为建议模式，规避全自动调度在物理极限下引发的雪崩效应。

6. 跨班次时间窗断裂与工单挂起
- 澄清点：若长耗时工单在临近交接班时触发，是强制派发给当前班次人员，还是推入缓冲队列交接给下一班次？
- 架构考量：影响调度算法的时间窗约束。需在核心算子中加入“剩余排班时间 > 预估工时”校验，并支持工单状态机的交接状态转移。

7. 边缘弱网环境的状态一致性
- 澄清点：地下车库等核心设备区通常存在信号盲区，系统如何保障下发给现场工人的 P0 抢修指令或现场上报数据不丢失？
- 架构考量：移动端是否需要采用离线优先架构，支持本地持久化与断网重传以保证幂等性；同时高优指令不可单点依赖 App 推送，是否需要建立包含短信、语音外呼的异构多级触达机制。
