---

## 1. 系统目标

为星汇智慧产业园构建一套全自动智能调度系统，通过大模型理解所有输入事件并自动派发给最合适的运维人员或机器人执行，实现"零人工调度"，大幅降低人力成本。

目标指标：
- 自动化率 > 95%
- 平均响应时间 < 3 分钟
- 调度主管可以转型去做更有价值的工作

---

## 2. 整体架构

```
 传感器告警 ─┐
 摄像头识别 ─┤                                      ┌─> 工人 App
 业主投诉  ─┤──> 事件接入网关 ──> LLM 核心 ──>派单 ──┤
 巡检报告  ─┘                                      └─> 机器人调度接口
```

采用微服务架构。各组件通过 REST API 通信，核心语言 Python + FastAPI。部署在 Kubernetes 集群。

---

## 3. 核心流程

当任何来源的事件进入系统后，立即调用 Qwen-Plus-3.5 进行理解和决策。Prompt 模板如下：

```
你是一个园区调度专家。现在有一个事件：
{event_text}

请按以下 JSON 格式输出：
{
  "priority": "P0-P3",
  "required_skills": ["..."],
  "suggested_actions": "...",
  "estimated_duration_min": N
}
```

LLM 返回结果后，系统直接根据 `priority` 决定是否插队派单，根据 `required_skills` 匹配最近的工人。整个链路无需人工参与，真正做到"智能调度"。

---

## 4. 数据流

所有来源的告警、视觉识别和投诉进入系统后，**1 : 1 生成工单**，立即进入派单队列。这样能保证零丢失、零延迟，每一个信号都被重视。

对于重复投诉（同一事件多人反映），我们用 `complaint_count` 字段累计，作为优先级的加权因子（见第 6 节）。

---

## 5. 状态机

工单有以下状态：

- `CREATED` — 刚创建
- `ASSIGNED` — 已派给某个执行单元
- `IN_PROGRESS` — 执行中
- `COMPLETED` — 已完成

状态转移通过 FastAPI 接口触发，由执行人员在 App 上点击按钮驱动。例如工人点击"开始处理"后状态从 `ASSIGNED` 变为 `IN_PROGRESS`。

---

## 6. 优先级算法

优先级计算公式：

```python
def calc_priority(event):
    return (
        event.complaint_count * 10
        + event.severity
        + event.age_minutes / 60
    )
```

`complaint_count` 是该事件被反映的次数，被反映得越多说明越重要。`severity` 取自 LLM 返回的 P0-P3 映射（P0=10, P1=7, P2=4, P3=1）。`age_minutes` 是事件已经等待的分钟数，防止新事件一直插队导致旧事件饿死。

---

## 7. 调度策略

采用贪心算法：每个事件进入队列后，按优先级从高到低遍历，为每个事件找到"距离最近 + 技能匹配 + 空闲中"的执行单元。

```python
def dispatch(event, candidates):
    """
    统一的派单函数，工人和机器人共用
    candidates: list of (worker_or_robot, distance, skill_match)
    """
    feasible = [c for c in candidates if c.idle and c.skill_match]
    if not feasible:
        return None
    return min(feasible, key=lambda c: c.distance)
```

工人和机器人都抽象为"执行单元（ExecutionUnit）"，使用同一个 `dispatch()` 函数。这样系统非常简洁，也方便未来扩展到更多种类的执行单元（比如无人机）。

---

## 8. LLM 使用策略

全链路使用 **Qwen-Plus-3.5**，效果最好。每个事件平均调用 2-3 次 LLM：

1. **初始理解与决策**（Section 3 的主调用）
2. **生成派单指令文本**（给工人 App 发送人类可读的任务描述）
3. **处置完成后的总结报告**（便于存档和后续复盘）

成本方面，按照每月 15 万事件估算，月成本在可接受范围内。业务价值远大于 API 成本。

---

## 9. 降级方案

- **LLM 调用失败**：最多重试 3 次（指数退避），失败后发送告警邮件给调度主管
- **派单失败**（所有候选执行单元都不可用）：保存到失败队列，每 10 分钟重试一次
- **下游 API 失败**：记录日志，等待 SRE 排查

---

## 10. 测试策略

- **单元测试**：对每个模块做单元测试，LLM 调用全部 mock 成固定返回值，pytest 覆盖率目标 80%
- **集成测试**：搭建 staging 环境，由测试人员手工触发几种典型事件，观察端到端流程

---

## 11. 部署与监控

- 部署在 Kubernetes，3 副本保证高可用
- 日志通过 ELK 收集
- 指标通过 Prometheus + Grafana 监控
